# 单一场景数据源方案

## 目标

把当前分散在多处的场景定义，收敛成一份唯一的权威数据源，避免下面这几类问题继续扩大：

- `data.py` 和 `js/game-scenes.js` 各自维护一套剧情，容易不同步。
- `文本/*.txt` 通过正则生成场景，容错过多，错误会被静默吞掉。
- `data/scenes.dsl.json` 已存在，但当前没有真正成为运行时唯一来源。
- 测试只覆盖 Python 场景树，网页实际运行内容和测试对象并不完全一致。

## 结论

建议把 [scenes.dsl.json](/F:/PYTHON%20%20ZUOYE/文字冒险-解谜/data/scenes.dsl.json) 定为唯一权威场景源。

原因：

- 它已经是结构化数据，适合校验、测试、构建、版本管理。
- 它可以同时服务 Python 版和 Web 版，不需要再维护两套场景定义。
- 比直接把 `文本/*.txt` 作为长期权威源更稳，因为 `txt` 目前更像策划草稿，格式不稳定、重复定义多、解析规则复杂。
- 比继续把 `js/game-scenes.js` 当源文件更好，因为 JS 文件里既有数据又有逻辑，而且已经出现大量重复 ID 覆盖。

## 未来结构

收敛后建议是这套职责分层：

- `data/scenes.dsl.json`
  - 唯一权威场景数据源
  - 只保存场景、选项、条件、效果、资源引用等声明式数据
- `data/hints.json`
  - 提示数据
  - 可继续单独维护
- `runtime/scene_logic.py`
  - Python 版条件/效果注册表
- `js/scene-logic.js`
  - Web 版条件/效果注册表
- `engine.py`
  - Python 运行时引擎
  - 只负责加载 JSON、执行注册表逻辑、管理状态
- `js/game-engine.js`
  - Web 运行时引擎
  - 只负责加载 JSON、渲染、执行注册表逻辑、管理状态

建议退役或降级为迁移工具的文件：

- [data.py](/F:/PYTHON%20%20ZUOYE/文字冒险-解谜/data.py)
- [js/game-scenes.js](/F:/PYTHON%20%20ZUOYE/文字冒险-解谜/js/game-scenes.js)
- [build_js.py](/F:/PYTHON%20%20ZUOYE/文字冒险-解谜/build_js.py)

## 数据边界

### 1. 哪些内容必须进场景源

- 场景 `id`
- 场景正文 `desc`
- 选项列表 `options`
- 场景进入效果 `on_enter`
- 条件表达式 `cond`
- 效果表达式 `effs`
- 背景图、人物图、结局图等资源引用
- 场景标签，如 `room`、`ending`、`trap`、`side_story`

### 2. 哪些内容不直接写死在场景源里

- 原生 Python lambda
- 原生 JS 函数体
- DOM 操作细节
- 本地存档 API 调用

这些应该变成“注册表里的可复用动作”，由场景数据用名字去引用。

## 建议场景 Schema

建议把当前 JSON 逐步统一成下面这种结构：

```json
{
  "hall_main": {
    "meta": {
      "room": "hall",
      "background": "images/backgrounds/hall_main.webp",
      "tags": ["hub"]
    },
    "desc": "你站在大厅中央……",
    "on_enter": [
      { "run": "refresh_hall_desc" },
      { "run": "trigger_butler_side_story", "when": ["medal_count_gte:3", "not_flag:side_butler_triggered"] }
    ],
    "options": [
      { "text": "前往图书馆", "target": "library_entry" },
      { "text": "开启中央密室", "target": "final_chamber_entry", "when": ["medal_count_gte:7"] }
    ]
  }
}
```

### 条件表达式建议

统一使用短语法，不再在 JSON 里混杂 JS/Python 代码：

- `has_item:怀表`
- `not_item:怀表`
- `has_clue:烧焦的纸片`
- `flag:side_butler_completed`
- `not_flag:side_butler_triggered`
- `medal_count_gte:7`
- `all_flags:side_butler_completed|side_painting_completed`

### 效果表达式建议

- `gain_item:怀表`
- `remove_item:机械齿轮`
- `gain_clue:前世记忆`
- `set_flag:side_butler_triggered`
- `clear_flag:side_butler_triggered`
- `add_medal:1`
- `mark_ending:七重谜语的真相`
- `redirect:side_story_1_start`

## 运行时设计

### Python 版

- `engine.py` 启动时加载 `data/scenes.dsl.json`
- 把每个场景转成 `Scene`/`Option` 对象，或直接保持字典
- 条件和效果通过注册表解释执行
- 不再从 [data.py](/F:/PYTHON%20%20ZUOYE/文字冒险-解谜/data.py) 手写构造场景

### Web 版

- `index.html` 启动后通过 `fetch("data/scenes.dsl.json")` 读取场景数据
- `js/game-engine.js` 只做渲染和状态流转
- 场景触发逻辑通过 `scene-logic.js` 解释执行
- 不再依赖生成后的 [game-scenes.js](/F:/PYTHON%20%20ZUOYE/文字冒险-解谜/js/game-scenes.js)

## 迁移策略

分四步走，风险最低。

### 第一阶段：先确立 JSON 为准

- 继续保留现有文件，但明确以后只改 `data/scenes.dsl.json`
- 新增校验脚本，检查：
  - 场景 ID 是否重复
  - `target` 是否存在
  - 条件/效果语法是否合法
  - 引用资源文件是否存在
- 让 CI 先基于 JSON 做校验

### 第二阶段：Python 改为读 JSON

- 新增 `load_scenes_from_json()`，让 Python 版直接从 `scenes.dsl.json` 构建场景
- 保留旧 `build_scenes()` 一小段时间用于对照
- 测试切到 JSON 加载路径

### 第三阶段：Web 改为读 JSON

- 去掉 `js/game-scenes.js` 的场景定义职责
- 页面启动时改为加载 `data/scenes.dsl.json`
- 把 `backgroundMap`、结局资源映射逐步并入场景 `meta`

### 第四阶段：退役旧生成链

- 停止维护 `build_js.py`
- 停止维护 `build_js_all.py`
- `文本/*.txt` 改成纯策划文档或一次性导入源
- `data.py` 只保留状态结构和引擎辅助代码，不再保留剧情定义

## 对现有项目最重要的几条约束

### 1. 不再允许“同名场景重复赋值”

现在 [game-scenes.js](/F:/PYTHON%20%20ZUOYE/文字冒险-解谜/js/game-scenes.js) 已经出现大量同名 ID 重复定义。迁移后必须做到：

- 同一个 `scene_id` 只能在 `scenes.dsl.json` 中出现一次
- 构建校验一旦发现重复，直接失败

### 2. 不再允许“缺 target 自动跳大厅”

现在生成器默认把缺目标选项兜底到 `hall_main`，这会把文本错误伪装成正常剧情。迁移后：

- 缺失 `target` 直接校验失败
- 不再自动补“返回大厅”
- 需要返回大厅时，必须显式写在场景数据里

### 3. 不再允许运行时猜测场景

现在前端有 `inferSceneTarget()` 这种模糊猜测逻辑。迁移后：

- 目标场景不存在就报错
- 构建期先拦截
- 运行时只保留极简的错误提示，不做智能猜测

### 4. 统一状态字段

目前 Python 和 Web 的状态字段也有风格差异：

- Python：`current_scene_id`
- Web：`currentSceneId`

建议统一成一套序列化字段，例如全部使用 snake_case：

- `current_scene_id`
- `hall_medal_count`
- `run_started_at`
- `visited_options`

Web 侧内部可以转 camelCase，但存档和 JSON 结构尽量统一。

## 文本文件怎么处理

`文本/*.txt` 不建议继续作为长期权威源，但也不需要立刻删除。

建议：

- 短期：保留为策划稿
- 中期：写一个一次性导入脚本，把可解析内容导入 JSON
- 长期：要么完全停止使用，要么把它们限制成“只读设计文档”

也就是说：

- 剧情运行依赖 `scenes.dsl.json`
- 策划讨论可以继续写 `txt`
- 但 `txt` 不再直接决定游戏实际内容

## 验收标准

当下面这些条件都满足时，说明“单一场景数据源”真正落地了：

- Python 版和 Web 版都从 [scenes.dsl.json](/F:/PYTHON%20%20ZUOYE/文字冒险-解谜/data/scenes.dsl.json) 读取场景
- [data.py](/F:/PYTHON%20%20ZUOYE/文字冒险-解谜/data.py) 不再手写剧情树
- [game-scenes.js](/F:/PYTHON%20%20ZUOYE/文字冒险-解谜/js/game-scenes.js) 不再保存场景数据
- 删除 `inferSceneTarget()` 这类运行时猜测逻辑
- 校验脚本能阻止重复 ID、坏 target、坏资源路径进入主分支
- 测试覆盖对象和网页实际运行对象是同一份场景数据

## 推荐实施顺序

建议按这个顺序做，成本最低：

1. 先补一套 `scenes.dsl.json` 校验脚本。
2. 再让 Python 版从 JSON 读场景。
3. 再让 Web 版从 JSON 读场景。
4. 最后清理旧脚本和重复定义。

## 下一步可直接开工的事项

如果按这个方案继续推进，下一步最值得先做的是：

1. 新增 `validate_scenes.py`
2. 给 `scenes.dsl.json` 补统一 schema
3. 把 Python 版切到 JSON 加载

这三步做完，项目结构会立刻清晰很多，后面再改剧情和 UI 都会轻松不少。
