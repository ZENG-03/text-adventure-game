谜题与解谜系统架构
1. 核心解谜逻辑理念
解谜不再是简单的“找钥匙开锁”，而是多重维度的“信息重组”与“多条件触发”。解谜的核心不仅在于推进进度，更是挖掘剧情（获取“真相线索”）。

A. 多条件交叉解谜体系
传统的解谜仅判断 hasItem("Key")。在现在的体系中，解谜需要综合：

物品 (Items)：拾取到的实体道具（例如生锈的钥匙、特殊的宝石）。

线索 (Clues)：调查日记、暗号得到的纯知识型线索。

前世记忆 (Profile Visited)：上一周目玩家做过的决策（memory["known_codes"]）。

机关状态 (Flags)：比如发电机是否开启，电闸是否接通（使用 state["scene_states"] 追踪）。

伏笔联动：如果玩家已经注意到某个异常（foreshadow_seen 包含特定 key），则直接给出提示或跳过步骤。

只有这些条件形成逻辑链闭环，才能解开核心锁。

C. 解谜的黄金节奏模型
每个主要谜题必须遵循：

探索 → 小奖励 → 解谜 → 剧情 → 强反馈 → 再探索

示例（音乐室）：

探索：推开门，看到管风琴和钢琴

小奖励：钢琴盖内发现“七音提示”

解谜：输入音符顺序

剧情：听见女人哼唱“他始终没有听完这首曲子……”

强反馈：暗格打开，获得【旋律徽章】+ 调音扳手

再探索：钢琴内部还有未发现的机关

D. UI 提交流程互动
物品提交系统：允许分步提交多个物品（如中央雕像需要 3 块宝石）。

提交校验：比对所需物品列表，成功后跳转到解锁场景。

与阿斯特状态联动：若阿斯特阶段 ≥ 2，提交可能被打断或增加额外条件。

2. 解谜的伏笔与记忆联动
A. 伏笔触发逻辑
每个场景可定义伏笔列表，包含条件和效果：

python

        "condition": lambda s: "photo_same_face" not in s["foreshadow_seen"],
        "effect": lambda s: s["foreshadow_seen"].append("photo_same_face")
    }
]

3. 存档数据与记录验证（底层支持）
物品与线索 (inventory, clues)：单周目重置，保证解谜路线需要重走，但在二周目体验中可引入“捷径”。

场景状态 (scene_states)：记录每个场景的本地进度（如箱子已打开、照片已查看）。

伏笔记录 (foreshadow_seen)：跨场景持久化，影响阿斯特阶段和解谜提示。

成就与徽章：记录区域是否已破解，部分解谜成功后防止重复触发。

提示机制：若玩家卡关，随着交互次数增加，提示逐渐明显（例如第一次“似乎需要密码”，第二次“工具排列有规律”，第三次直接高亮数字）。

4. 商业级解谜设计 Checklist
每个谜题都遵循“黄金节奏模型”

有至少一个“机制型谜题”（如视线、排序、多层）

解谜失败不挫败（不用“错误”，改用“你感觉这个思路接近了……”）

二周目提供捷径（记忆密码、跳过部分收集）

解谜成功后给予强反馈（音效、特效、徽章、剧情）

与阿斯特状态联动（高警惕时解谜更困难或出现干扰）

存在隐藏解法（不同输入触发不同结果或隐藏剧情）
